## 1. 調査対象の概要と本設計との立ち位置の違い

nashsu/llm_wiki は Karpathy 型 LLM Wiki(本設計の先行アプローチ (a))を実装した
デスクトップアプリ(Tauri v2、TypeScript + Rust、GPL-3.0)。「個人的な知識ベースが
自動で構築される」ことを目指し、文書(PDF/DOCX 等)を Ingest すると LLM が
永続 Wiki を段階的に構築・保守する。

| 観点 | nashsu/llm_wiki | 本設計 |
|---|---|---|
| 形態 | 専用デスクトップアプリ | ディレクトリ一式のコピー(R1) |
| 一次情報 | 投入された外部文書(raw/sources/) | プロジェクトの実体(コード・設定) |
| 更新契機 | Ingest 時の自動処理+フォルダ監視 | 人間の手動トリガーのみ |
| 腐敗対策 | レビューキュー・キャッシュ | 承認ゲート・寿命分離・verified_against |

形態と自動化方針は正反対だが、「LLM に知識を繋がせて総合する」という中核課題は
同一であり、その解法(§2)には流用できる発想が多い。なお三層構造
(Raw sources / Wiki / Schema)と Lint・log.md・Obsidian vault 互換は
本設計が先行アプローチ検討で既に把握済みの要素であり、本レポートでは差分のみ扱う。

## 2. 参考になる機能(本設計への応用案つき)

優先度: ◎=次の改訂で採用を検討 / ○=実運用で課題が出たら採用 / △=発想のみ参考

### 2.1 ◎ purpose.md — 「構造」と「方向」の分離

- **先方の仕組み**: Schema 層を `schema.md`(構造ルール)と `purpose.md`(目的・
  主要な問い・調査スコープ・発展中のテーゼ)に分業。LLM は Ingest・Query のたびに
  purpose.md を読んで文脈を得る。使用パターンに基づき LLM が purpose の更新を
  提案することもできる
- **本設計の現状**: WIKI_RULES.md は構造・手順のみで、「このプロジェクトで何を
  蓄積したいか・何が主要な問いか」を書く場所がない。Synthesize は目的を知らないまま
  横断分析することになる
- **応用案**: テンプレートに `01-Purpose.md`(人間が書く・LLM は読むだけ、
  更新提案は可)を追加するか、00-Index.md 冒頭に「目的と主要な問い」節を設ける。
  特に Synthesize の観点(ギャップ・新仮説)は purpose があると精度が大きく
  変わるはず。コスト最小で効果が見込める

### 2.2 ◎ 二段階処理(分析 → 生成)

- **先方の仕組み**: Ingest を「①分析: 主要エンティティ・概念・論点の抽出、
  既存 Wiki との接続・矛盾・推奨構造の整理 → ②生成: ページ作成・更新」の
  2 段階に分け、「significantly better quality」を得たとする
- **応用案**: Record と Sync の手順に「書く前の分析ステップ」を明文化する。
  Record なら「新規ページを書く前に、既存ページとの接続・矛盾を列挙せよ」の
  1 行を WIKI_RULES 5.2 に足すだけで移植できる。現行手順の「同じ主題の既存ページが
  ないか確認」を一般化した形

### 2.3 ◎ コンテンツハッシュによる増分スキップ

- **先方の仕組み**: ソースの SHA256 を Ingest 前に計算し、不変なら自動スキップ。
  キューは永続化されアプリ再起動後も有効。トークン消費と処理時間を削減
- **本設計の現状**: `verified_against.version` は git ハッシュまたは mtime。
  mtime は touch やコピーで変わるため偽陽性(不要な再検証)を生む
- **応用案**: git 非管理プロジェクトでは mtime の代わりに(または併記で)
  `sha256sum` の先頭 12 桁程度を記録する。sha256sum はどの環境にもあり R1 を
  損なわない。Sync 手順3に「ハッシュ一致なら本文照合を省略してよい」を追記すると、
  大規模プロジェクトの Sync 負荷(未決事項)への部分回答にもなる

### 2.4 ○ 提案への定型アクション付与(非同期レビューキュー)

- **先方の仕組み**: Ingest 中に LLM がフラグを立てた項目はレビューキューに入り、
  アクションは **Create Page / Deep Research / Skip の定型 3 種に限定**。
  LLM の恣意的なアクション生成を防ぎ、Ingest 本体をブロックしない
- **本設計の現状**: 「未収載の実体」「Lint 報告」「Synthesize の関連性提案」は
  いずれも人間の承認待ちになるが、提案の書式が自由で、人間が判断・指示に
  手間がかかる(放置されやすい)
- **応用案**: 承認待ち項目に定型アクションを付ける書式を規約化する。例:
  `- [ ] src/new_module.py 未収載 → 提案: ページ化 / 調査して Record / 無視`。
  人間は選ぶだけでよくなり、承認ゲート運用の摩擦(本設計の腐敗対策が抱える
  コスト)を下げる。チェックボックス形式なら「どれを承認したか」が
  ファイル上に残り、次の Promote が機械的に拾える

### 2.5 ○ 関連度の判定規準の明文化(4信号モデル)

- **先方の仕組み**: ページ間関連度を4信号の重み付き和で計算 —
  直接リンク(×3.0)、**ソース重複(×4.0)**、Adamic-Adar=共通隣接(×1.5)、
  タイプ親和性(×1.0)。最大の重みは「同じソースに由来するか」
- **応用案**: 本設計では数値計算はしないが、Synthesize の「関連性の提案」の
  判定規準として WIKI_RULES に列挙できる: ①同じ実体を `verified_against.target`
  に持つ(=ソース重複。最有力)、②共通ページから多くリンクされている、
  ③タグが重なる、④本文が相互言及。LLM の主観頼みだった提案に根拠の型を与え、
  提案時に「根拠: 同一ソース(src/config.py)」と書かせれば人間の承認判断も速くなる

### 2.6 ○ 構造由来のギャップ検出(グラフインサイト)

- **先方の仕組み**: グラフ構造から3種の洞察を機械抽出 —
  予期しない接続(コミュニティ横断エッジ、タイプ横断リンク、周辺↔ハブ結合)、
  知識ギャップ(次数≤1 の孤立ページ、疎なコミュニティ、ブリッジノード)、
  Louvain クラスタリング(凝集度 < 0.15 で警告)
- **応用案**: 数値化・クラスタリングは本設計の規模では過剰だが、**観点**は
  Synthesize 手順2に足せる: 「リンク・タグの構造から見て、孤立している主題、
  1 ページだけが橋渡ししている主題を指摘せよ」。内容ベースのギャップ検出
  (現行)に構造ベースの視点が加わる。Lint の孤立ページ検出(既存)とは
  「報告(Lint)/洞察と次の問いの生成(Synthesize)」で役割が分かれる

### 2.7 △ ギャップから「次の調査の問い」を生成する

- **先方の仕組み**: 知識ギャップ検出時に Deep Research(Web 検索)クエリを
  自動生成し、purpose.md + overview.md を読んでドメイン特化のトピックを作る
- **応用案**: Web 検索(Ingest 系)は本設計の対象外だが、「ギャップを
  **実行可能な問いの形**で出力する」点は Synthesize レポートの改善に使える。
  「未検証領域」を名詞の列挙でなく「次に Record すべき調査の問い」として
  書かせると、レポートが次の行動に直結する

### 2.8 △ ページタクソノミーの分割先候補

- **先方の仕組み**: `entities / concepts / sources / queries / synthesis / comparisons`
  の 6 分類。全ページに YAML frontmatter(type, title, sources[])
- **応用案**: 本設計の 3 分類(concepts / investigations / queries)で開始するのは
  変えないが、将来 concepts が肥大化した際の分割先として `comparisons`
  (対案比較)は筋が良い。`entities` は個人知識ベース向きでプロジェクト Wiki には
  不要。frontmatter に sources[] を持たせて出典追跡する発想は
  `verified_against.target` で実現済み

### 2.9 △ スキル化の先行実装(llm_wiki_skill)

- **先方の仕組み**: エージェント(Claude Code 等)向けスキルを別リポジトリで配布
  (`npx skills add ...`)。ドキュメントのみで SDK・ビルド不要。
  **トリガーは「LLM Wiki への明示言及時のみ」に意図的に限定**し、一般的な検索では
  発動しない設計
- **応用案**: 本設計の未決事項2(スキル化)の直接の先例。採用する場合は
  ①テンプレート本体と分離して配布する、②トリガーを「ai_wiki への明示言及時のみ」に
  限定する(本 README の推奨指示形式「ai_wiki の <操作名>」と整合する)、の
  2 点をそのまま踏襲できる

## 3. 参考にしない機能(理由)

| 機能 | 理由 |
|---|---|
| Ingest(外部文書取込)・フォルダ監視・自動更新 | 本設計の中核方針(実体が唯一のソース・手動トリガー)と正面から矛盾。自動増殖は腐敗リスクの主因として設計段階で排除済み |
| ベクトル検索(LanceDB)・トークン化検索 | 指示により対象外。かつ本設計はアプリを持たないため実装場所がない |
| UI(グラフビュー・エディタ・チャット画面等) | 指示により対象外。グラフ可視化は Obsidian オプションで代替可能 |
| Deep Research(Web 検索統合) | 外部ソース取込の一種。§2.7 の「問いの形にする」発想のみ吸収 |
| ローカル HTTP API + MCP サーバー | 常駐アプリが前提で R1(コピーで導入)と逆方向。ただし将来「複数プロジェクトの ai_wiki を横断検索したい」要求が出た場合の実現形として記憶に値する(ファイルベースのまま MCP サーバーだけ被せる形は可能) |
| Web クリッパー(Chrome 拡張) | 個人知識ベース向け機能 |

## 4. 設計ドキュメント(design_llm_wiki.md)への影響

- **未決事項2(スキル化)**: llm_wiki_skill の「明示言及のみトリガー・本体と分離配布」を
  採用時の設計指針として追記する価値あり(§2.9)
- **未決事項5(Synthesize 負荷)・Sync 粒度**: コンテンツハッシュによるスキップ(§2.3)が
  部分回答になる
- **未決事項6(関連性提案の精度)**: 判定規準の明文化(§2.5)と定型アクション書式(§2.4)が
  承認負荷の軽減策になる
- **新規候補**: purpose ファイルの追加(§2.1)と Record の分析ステップ(§2.2)は
  未決事項に挙がっていない新しい改善点。次回のテンプレート改訂で検討する
