# 汎用LLM Wiki 検討ドキュメント

- 作成日: 2026-07-12
- 更新日: 2026-07-12
- ステータス: 検討中(第2版。初版に design_llm_wiki_another1.md の知見を選択的に統合)
- 対象: どのプロジェクトにも配置可能な、LLM が管理するプロジェクト内 Wiki の設計

---

## 1. 背景と目的

### 1.1 背景

LLM エージェント(Claude Code 等)はセッション間で記憶を保持しない。プロジェクト固有の
知識・調査結果・設計判断をセッションをまたいで引き継ぐには、プロジェクト内に
「LLM が読み書きするローカルナレッジベース」を置くのが有効である。

### 1.2 本設計の根本的な目的

LLM を用いた検証・調査・思索の結果は、現状ではプロジェクト配下に**乱雑に散らばっている**
(コードコメント、調査メモ、チャットログ、設計ドキュメントなど)。この状態では:

- 過去に同じ問いを検証済みかどうかの把握が困難(**再調査の浪費**)
- 個別の知見が独立して存在し、相互の関連性が見えにくい(**知識のサイロ化**)
- 複数の調査結果を総合することで初めて見える**新たな課題や手法**に気づきにくい

本 Wiki は、散在する知見を「関連性で繋がった知識の地図」として再構築することを
第一の目的とし、具体的には次の3つを実現する:

1. **地図化(マッピング)**: 過去の検証・調査・思索を一元的に可視化し、全体像を俯瞰できるようにする
2. **関連性の追跡(トレーサビリティ)**: ページ間のリンクを辿ることで、ある知見が
   別のどの知見と接続しているかを理解できるようにする
3. **総合による新洞察(シンセシス)**: 個々の知見を横断的に統合し、これまで気づかなかった
   パターン・矛盾・トレンド・未検証領域を**能動的に抽出**する仕組みを提供する

### 1.3 参考にした先行アプローチ

**(a) Karpathy 型 LLM Wiki**
(参考: https://zenn.dev/tsurubee/articles/llm-wiki-connecting-knowledge)

- 三層構造: Raw sources(人間が追加・不変)/ Wiki(LLM が管理)/ Schema(人間が定義する規約)
- 三操作: **Ingest**(ソース投入→関連ページ自動更新)/ **Query**(検索と回答の資産化)/ **Lint**(矛盾・孤立ページの定期検出)
- 最大の価値は要約ではなく「複数ソースの自動統合(繋げる力)」。
  LLM は飽きず忘れず、1パスで多数ファイルのクロスリファレンスを更新できる
- 課題: 自動増殖ゆえの**情報腐敗**(誤情報・重複・人間の認識とのズレ)

**(b) 手動トリガー型 ai_wiki**
(参考: https://qiita.com/usayamadausako/items/c8b5cca97554f6f64782)

- 自動増殖を廃し、**人間が節目で指示したときのみ** LLM が Wiki を更新
- フロー型(`investigations/` 一時情報)とストック型(`concepts/` 恒久情報)を寿命で分離
- 削除ではなく `archive/` への退避(段階的削除)でリンク切れと消失事故を防ぐ
- `00-Index.md` を唯一の入口にし、必要なページだけ読み込ませてコンテキストを節約
- 導入コストが低く、腐敗リスクは「中」に抑えられる

| 観点 | (a) Karpathy型 | (b) 手動トリガー型 |
|---|---|---|
| 更新契機 | Ingest 時に自動連鎖更新 | 人間の指示のみ |
| 整理の主体 | LLM(Lint) | 人間の判断 + LLM 実行 |
| 導入コスト | 高 | 低 |
| 腐敗リスク | 高 | 中 |
| 強み | ソース横断の自動統合 | 人間の認識との同期維持 |

本設計は **(b) をベースに、(a) の Lint(整合性チェック)と Index/Log の規律を取り込み、
さらに 1.2 の目的(地図化・関連性追跡・総合洞察)を実現する Synthesize 操作を独自拡張として加える**。
ただし (a) の課題である情報腐敗を再導入しないため、関連性の生成は「自動反映」ではなく
「提案 → 人間承認 → 反映」のフローに限定する(3.8 参照)。

## 2. 要件

本 Wiki が満たすべき要件は次の3点。

| # | 要件 | 含意 |
|---|---|---|
| R1 | **どのプロジェクトにも配置可能** | 特定言語・特定リポジトリ構成・特定ツールに依存しない。導入はディレクトリ一式のコピーで完結する |
| R2 | **Wiki 用ディレクトリ配下のみを更新** | LLM が Wiki 運用の名目でプロジェクト本体のコード・ドキュメントを書き換えない。書き込み境界を規約だけでなく仕組みで担保したい |
| R3 | **指示されたときにプロジェクトの状態をチェックし Wiki を更新** | 更新契機は人間の明示的な指示(手動トリガー)。更新時は Wiki 内の記述を鵜呑みにせず、プロジェクトの実状態(コード・設定・ドキュメント)と突き合わせて反映する |

R3 の「状態をチェックし」が本設計の独自点で、(b) が「セッションの調査結果を書き残す」
ことを主眼とするのに対し、本 Wiki は**プロジェクトの実状態が一次情報**であり、
Wiki は常にそこから再導出・検証可能な二次情報である、という位置づけを取る。

## 3. 全体設計

### 3.1 位置づけ(何であって何でないか)

- Wiki は**二次情報**。一次情報はプロジェクトの実体(コード・設定・正式ドキュメント)。
  矛盾したら実体が正で、Wiki を直す
- Wiki は**プロジェクト本体の正式ドキュメント(README 等)の代替ではない**。
  正式ドキュメントに昇格すべき内容が生まれたら、人間がレビューして本体側へ転記する
  (LLM は R2 により本体を書けないので、この昇格は必然的に人間のゲートを通る)
- 解読・実装そのものの作業場ではない。作業はプロジェクト側で行い、**知見・思索・調査結果だけ**を
  相互に繋がった形で Wiki に蓄積する

### 3.2 ディレクトリ構造(案)

```
<プロジェクトルート>/
└── ai_wiki/
    ├── WIKI_RULES.md        # スキーマ層: 運用規約(人間が定義、LLMは読むだけ)
    ├── 00-Index.md          # 入口: 全ページの目録 + 1行要約 + 関係性マップ + タグ一覧
    ├── 01-Purpose.md        # 目的・主要な問い(人間が定義、LLMは読むだけ。更新提案は可)
    ├── 90-Log.md            # 時系列の更新ログ(いつ・何を・なぜ)
    ├── 99-Insights.md       # Synthesize の出力レポート(総合知見・日付セクション追記式)
    ├── concepts/            # ストック型: 恒久的な知識(アーキテクチャ、設計判断、用語)
    ├── investigations/      # フロー型: 調査中・作業中の一時メモ
    │   └── archive/         # 退避場所(削除の代わりにここへ移動、日付プレフィックス付与)
    └── queries/             # 質問と回答の資産化(任意運用)
```

設計意図:

- **`ai_wiki/` 一階層に閉じる**: R2 の境界を「1つのディレクトリ配下」という最も単純な形で定義できる
- **命名が `a` で始まる**: エクスプローラで最上部に表示され発見しやすい((b) の知見)
- **WIKI_RULES.md を Wiki 内に同梱**: プロジェクト側の CLAUDE.md に長文規約を書かずに済み、
  コピーだけで規約ごと移設できる(R1)。CLAUDE.md には参照1行だけ追記する
- **concepts / investigations の寿命分離**: 腐敗対策の中核。「古くなって困る情報」を
  investigations に隔離し、concepts には検証済みの安定知識のみ昇格させる
- **`99-Insights.md` の新設**: Synthesize(3.8)の出力先を明確にし、「総合による新しい気づき」を
  Wiki 内の一級の成果物として扱う
- **`01-Purpose.md` の新設**: 規約(WIKI_RULES = 構造)と目的(Purpose = 方向)の分離。
  「この Wiki が何を知ろうとしているか」を人間が宣言し、Record の取捨選択と Synthesize の
  分析に文脈を与える。LLM は編集禁止だが、Synthesize で更新を提案できる
  (nashsu/llm_wiki 調査 §2.1 由来。→ docs/survey_nashsu_llm_wiki.md)
- **`queries/` は任意**: (a) の Query 資産化は有用だが必須ではない。使わないプロジェクトでは
  空のまま置くか作らない

### 3.3 ページ種別とスキーマ

各ページは YAML フロントマターで最小限のメタデータを持つ:

```markdown
---
type: concept | investigation | query
status: active | stale | archived
created: 2026-07-12
updated: 2026-07-12
verified_against:
  target: "src/config.py"          # 検証時に突き合わせた実体のパス(複数可)
  version: "abc123"                # git 管理下ならコミットハッシュ、なければファイルの mtime
  date: "2026-07-12"               # 検証実施日
tags: [認証, API設計]              # 横断的タグ(ディレクトリ階層を超えた関連性の表現)
topics: [認証フロー, セッション管理]  # investigation のみ: 含まれる独立した主題
question: "..."                    # type: query の場合に必須(問い)
answer: "..."                      # type: query の場合に必須(回答の要旨)
---
```

**ページ単位の原則**: `concepts/` は**主題型**(1概念 = 1ページ)に保ち、
セッション型(「◯月◯日の調査結果」)のページを置かない。知識の地図(リンク・
関係性マップ)が意味を持つのはページが主題を表すときだけだからである。
`investigations/` は調査単位(セッション型)でよく、主題型への再構成は
Promote が承認ゲート付きで行う(Record 時の自動分割を主手段にしないのは、
セッションごとの主題ページ自動生成が微妙に異なる名前・粒度の断片を増殖させ、
Karpathy 型の腐敗を再導入するため)。`topics` はこの再構成の判断材料になる。

- `verified_against` が R3 の「状態チェック」の痕跡になる。更新時に「いつ・何と突き合わせたか」を
  必ず残すことで、次回チェック時に鮮度を機械的に判定できる。git 管理下ではコミットハッシュ、
  非管理ならファイルの mtime を `version` に記録する
- `tags` はディレクトリ階層を超えた横断的関連の表現手段。付与は Record/Promote 時
  (ページを書く LLM 自身が付ける)とし、既存ページのタグを一括で書き換える操作は設けない
- ページ間リンクは相対パスの Markdown リンク(`[title](../concepts/xxx.md)`)。
  Obsidian の `[[wikilink]]` は Obsidian 依存になるため既定では使わない(R1)。
  Obsidian を使うプロジェクトでは WIKI_RULES.md で wikilink 許可に切り替えてよい
  (その場合はページ名の一意性を必須とし、グラフビューによる関係性の可視化が利用できる)

### 3.4 操作セット

(a) の三操作を手動トリガー前提に読み替え、Synthesize を加えた5操作とする。
すべて人間の指示で発動する。

| 操作 | 指示の例 | LLM がやること |
|---|---|---|
| **Sync**(中核・R3) | 「プロジェクトの状態をチェックして Wiki を更新して」 | ①Index と各ページの `verified_against` を読む → ②実体(コード・設定・ドキュメント)と突き合わせ → ③乖離をページに反映、鮮度メタデータ更新 → ④Index と Log を更新 |
| **Record** | 「ここまでの調査結果を Wiki に記録して」 | **分析→生成の二段階**。①含まれる主題・既存ページとの接続・矛盾を列挙 → ②investigations/ に新規ページ作成(既定は1調査1ページ。独立した主題が複数なら分割可。実体で裏取りし、タグ・topics・関連リンクを付ける) |
| **Promote/Organize** | 「一区切りついたので Wiki を整理して」 | investigations の各ページを評価 → 恒久価値のあるものを **主題単位に再構成して** concepts へ昇格(分割・既存 concept への統合・単純移動のいずれかを提案し、承認後に実行)→ 用済みは archive へ移動(削除しない)→ リンク修正。承認済みの関連性提案(3.8)の反映もここで行う |
| **Lint** | 「Wiki の健全性をチェックして」 | 孤立ページ・リンク切れ・重複概念・矛盾・長期 stale・タグの乱れを検出し、**報告と提案まで**。修正は人間の承認後 |
| **Synthesize** | 「今までの概念・調査・Q&A を総合して、新たに見える知見を抽出して」 | 全 active ページを横断読込 → 共通パターン・矛盾・トレンド・ギャップ・新仮説を抽出 → `99-Insights.md` にレポート追記(3.8 参照)。既存ページの書き換えはしない |

3操作の責務境界:

- **Sync** = Wiki ↔ プロジェクト実体の整合(自動修正可。実体が正だから)
- **Lint** = Wiki 内部の構造品質チェック(検出・報告のみ。修正は人間承認後)
- **Synthesize** = Wiki 内の全知識の意味的な総合と新洞察の創出(出力先は `99-Insights.md` のみ)

### 3.5 書き込み境界の担保(R2)の方式検討

R2 をどの強度で担保するかは3案ある。

| 案 | 方法 | 強度 | 可搬性(R1) | 備考 |
|---|---|---|---|---|
| 案1 | WIKI_RULES.md の規約で「`ai_wiki/` 配下以外に書くな」と指示 | 弱(LLM の遵守頼み) | ◎ どの LLM エージェントでも通用 | 最低ライン。単独では事故を防げない |
| 案2 | Claude Code の permissions で `Edit`/`Write` を `ai_wiki/**` に限定 deny/allow | 強(ハーネスが強制) | △ Claude Code 専用。かつ Wiki 操作専用セッションでないと本体開発を阻害 | Wiki 更新を独立セッション・専用設定で走らせる運用なら有効 |
| 案3 | PreToolUse フックで書き込み先パスを検査し、Wiki 操作モード中は `ai_wiki/` 外への書き込みをブロック | 強 | △ Claude Code 専用。モード判定の仕組みが必要でセットアップが重い | (b) が「フック不要の簡潔さ」を利点に挙げた点と逆行 |

**推奨: 案1を必須の土台とし、案2をオプションとして提供する。**

- 通常運用では、Wiki 操作は「Sync して」等の明示指示で始まる短いタスクであり、
  その指示文脈下で規約違反の書き込みが起きる確率は低い。導入の軽さ(R1)を優先する
- 厳格運用したいプロジェクト向けに、`ai_wiki/` 内にオプション設定例
  (`optional/claude-settings-strict.json` として permissions 設定のサンプル)を同梱する
- 案3は導入コストが利益に見合わないため不採用(将来、事故が実際に起きたら再検討)

もう1つの境界として「**プロジェクト側からの読み取りは無制限、書き込みのみ制限**」を
明文化する。Sync(R3)は本体コードを読む必要があるため、読み取りまで制限してはいけない。

### 3.6 導入手順(R1)

1. 本リポジトリの `template/ai_wiki/` を対象プロジェクトのルートへコピーし、
   `ai_wiki/01-Purpose.md` に目的と主要な問いを記入する(数行でよい)
2. 対象プロジェクトの CLAUDE.md(または同等のエージェント指示ファイル)に1行追記:
   `ai_wiki/ は LLM 管理の Wiki。運用規約は ai_wiki/WIKI_RULES.md を参照。`
3. Wiki を git 管理するかを決める:
   - **管理する(既定推奨)**: セッション間・メンバー間で共有でき、履歴が残る
   - **管理しない**: 個人メモ扱いにしたい場合。`.git/info/exclude` か global gitignore に
     `ai_wiki/` を追加((b) はこちらを採用。リモートに個人の調査メモを上げたくないケース)
   - どちらでも動くよう、テンプレート側は git を前提にしない
4. 最初の指示: 「プロジェクトの状態をチェックして Wiki を初期化して(スコープ: README と主要設定)」
   → LLM が実体を読み、concepts に初期ページ(プロジェクト概要・構成)を数枚生成

手順1〜2 は手作業でも1分で終わるが、`install.sh <対象プロジェクトパス>` を用意して
コピー+追記を自動化してもよい(実装フェーズで判断)。

### 3.7 Sync(状態チェック)の動作設計(R3)

R3 の中核操作。指示を受けた LLM の手順を WIKI_RULES.md に規定する:

1. `00-Index.md` を読み、全ページの目録と `status`/`updated` を把握する
2. **人間が指定したスコープ**の実状態を把握する(指定がなければ既定スコープ:
   ディレクトリ構成、主要設定、README、直近変更ファイル。git 管理下なら前回 Sync 以降の
   `git log`/`git diff --stat` が最も効率的な差分源)。大規模プロジェクトでは
   「`src/auth/` 配下だけ Sync して」のようにスコープを絞った指示を推奨する
3. 各 active ページについて `verified_against` の対象と現状を突き合わせ:
   - 一致 → `updated`/`verified_against` のみ更新
   - 乖離 → 本文を実体に合わせて修正(実体が正)
   - 対象消滅 → `status: stale` にして Log に記録(archive 移動は Promote 時に人間判断)
4. 実体に存在するが Wiki 未収載の重要要素(新モジュール・新設定)があれば、
   **勝手に大量生成せず**、Index に「未収載」として列挙し、ページ化するかは人間に確認
5. `00-Index.md` と `90-Log.md` を更新(Log には日付・操作・変更ページ・理由を1行ずつ)

手順4が腐敗対策上の要点: Sync に自動増殖を持ち込まない。ページの新設は原則
Record/Promote(人間の意図がある操作)経由とする。同じ理由で、**ページ間の関連リンクの
一括生成・書き換えも Sync では行わない**(関連性の抽出は Synthesize の提案として出す。3.8)。

### 3.8 Synthesize(総合洞察)の動作設計

1.2 の目的3(総合による新洞察)を直接実現する操作。既存ページを書き換えず、
`99-Insights.md` へのレポート追記だけを行う読み取り中心の操作なので、
自動増殖による腐敗リスクを持ち込まない。

**LLM の実行手順(WIKI_RULES.md に規定)**:

1. `01-Purpose.md`(目的・主要な問い)を読んで分析の文脈とし、`00-Index.md` から
   全 active ページ(concepts・investigations・queries)の一覧を把握する。
   大規模な場合は人間がスコープを指定できる(「最近1ヶ月に更新されたページだけ」等)
2. 各ページの本文を読み込み、以下の観点で分析する:
   - **共通パターン**: 複数の調査で繰り返し登場するテーマや課題
   - **矛盾・対立**: 異なるページで食い違っている主張や前提
   - **トレンド**: 時間経過とともに変化している考え方(90-Log.md も参照)
   - **ギャップ(未検証領域)**: 言及されているが深掘りされていないトピック
   - **新たな仮説**: 複数の知見を組み合わせることで初めて導かれる示唆
   - **関連性の提案**: ページ間に張るべきなのに張られていないリンク
3. 抽出結果を `99-Insights.md` に**日付セクションとして追記**する(上書きしない)
4. レポート末尾に「この知見を元に Record(新規調査)や Promote(リンク反映・整理)を
   実行するかは人間が判断する」旨を付記する

**出力レポートの雛形**:

```markdown
## 実行日: 2026-07-12(対象: concepts 5件, investigations 3件, queries 2件)

### 共通パターン
- (複数ページに繰り返し現れるテーマと、その集約先の仮説)

### 矛盾・対立
- (ページ A とページ B で食い違っている記述の指摘)

### トレンド
- (Log から読み取れる関心・方針の変化)

### 未検証領域(ギャップ)
- (どのページにも書かれていないが必要そうなトピック)

### 新たな仮説・示唆
- (知見の組み合わせから導かれる提案。要検証マーク付き)

### 関連性の提案
- concepts/auth.md ←→ concepts/cache.md(依存関係。両者にリンク追加を提案)
```

**関連性の反映フロー(腐敗対策の要点)**: Synthesize が出す「関連性の提案」は提案止まりとする。
人間がレポートを見て承認したものだけを、次の Promote/Organize で各ページ末尾の
「関連ページ」セクションと Index の関係性マップ(3.9)に反映する。
これにより (a) 型の「LLM が誤った関連を自動で張り巡らせて腐敗する」リスクを、
人間のゲートで遮断しつつ、「繋げる力」の価値は確保する。

### 3.9 `00-Index.md` の構造

Index を単なるファイルリストではなく「知識の地図」として機能させるため、次の構造を標準とする:

```markdown
# プロジェクト知識地図 (Index)

## 関係性マップ
- 認証フロー ←→ セッション管理(実装依存)
- キャッシュ設計 ←→ パフォーマンス要件(トレードオフ)
(Synthesize の提案から人間が承認したものだけを掲載)

## タグ一覧
認証 / API設計 / デプロイ / テスト戦略

## ページ一覧
### concepts/(恒久知識)
- [アーキテクチャ概要](./concepts/arch.md) - 全体構造の説明
### investigations/(調査中)
- [セッションタイムアウト調査](./investigations/session_timeout.md) - 原因追求中
### queries/(Q&A・任意)
- [なぜJWTを採用したか](./queries/jwt.md) - 判断理由の記録

## 未収載の実体(Sync で検出)
- src/new_module.py - 新モジュール(ページ化するか要判断)
```

- ページ一覧の各行(ファイル名+1行要約)は Sync/Record/Promote が随時更新する
- 関係性マップの更新は 3.8 の承認フロー経由のみ。タグ一覧は各ページの `tags` の集約で、
  Promote 時に再集計する

### 3.10 コンテキスト効率

- LLM は毎回 Wiki 全体を読まない。**入口は常に `00-Index.md`**、そこから必要ページのみ読む
  (全ページを読むのは Synthesize のみで、これは人間が意図して発動する重い操作と位置づける)
- Index の各行は「ファイル名 + 1行要約」を保ち、Index 自体が肥大化(目安: 200行または50ページ超)
  したらディレクトリ単位のサブ Index に分割する
- 1ページの目安は 100〜200 行。超えたら分割を Lint が提案する

### 3.11 補足運用ルール

- **機密情報の禁止**: API キー・シークレット・個人情報を Wiki に記載しない(WIKI_RULES.md に明記)
- **archive/ への退避**: 日付プレフィックス(`2026-07-12_元ファイル名.md`)を付与して移動。
  archived ページは Lint・Synthesize の対象外
- **git コンフリクト**: Wiki を git 管理する場合、コンフリクト解決は必ず人間が行う
- **変更履歴**: ページ内に履歴テーブルは置かず、`90-Log.md` と git 履歴に任せる
- **多エージェント共存**: WIKI_RULES.md は特定エージェントに依存しない命令形の文面で書き、
  Claude Code 固有の設定(permissions 等)は `optional/` に分離する

## 5. 未決事項・今後の検討

1. **テンプレートの実装**: `template/ai_wiki/` 一式(WIKI_RULES.md 本文、Index/Log/Insights の雛形、
   フロントマター仕様、`optional/claude-settings-strict.json`)の作成。→ 次フェーズ
2. **Claude Code スキル化するか**: Sync/Record/Promote/Lint/Synthesize を `.claude/skills/` の
   スキルとして同梱すると指示が `/wiki-sync` 等で安定するが、Claude Code 依存が増す。
   規約ベース(WIKI_RULES.md に手順を書き、自然言語で指示)を既定とし、
   スキルはオプション同梱が妥当か。→ 実運用1プロジェクトで試してから判断
3. **install.sh の要否**: 手作業1分で済むため、複数プロジェクト展開が始まるまで保留
4. **`queries/` の要否**: 初版テンプレートに含めるか。含めても害はないが、
   使われない空ディレクトリは Lint のノイズになる。→ 初版では WIKI_RULES.md に
   「必要になったら作る」と書くに留める
5. **Synthesize の発動タイミング**: 完全手動を既定とするが、運用が乗ったら
   「Promote 後に必ず実行」等の慣行にするか検討。→ 実運用後に判断
6. **関連性提案の精度**: Synthesize が誤った関連を提案するリスクは承認ゲートで
   遮断されるが、提案のノイズが多いと承認負荷が上がる。→ 実運用で観察
