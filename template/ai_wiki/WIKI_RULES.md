# WIKI_RULES — ai_wiki 運用規約

この文書は、本ディレクトリ(`ai_wiki/`)を管理する LLM エージェントへの運用規約である。
Wiki に対する操作を指示されたら、まずこの文書を読み、以下の規約に従って作業せよ。
この文書自体は人間が管理する。LLM は WIKI_RULES.md を編集してはならない。

## 1. この Wiki の位置づけ

- この Wiki は**二次情報**である。一次情報はプロジェクトの実体(コード・設定・正式ドキュメント)。
  Wiki と実体が矛盾したら実体が正であり、Wiki 側を直せ
- この Wiki はプロジェクト正式ドキュメント(README 等)の代替ではない。正式ドキュメントへ
  昇格すべき内容を見つけたら、転記はせず「昇格候補」として人間に報告せよ
- 目的は3つ: ①過去の検証・調査・思索の**地図化**、②知見間の**関連性の追跡**、
  ③知見の総合による**新洞察の抽出**

## 2. 書き込み境界(最重要)

- **書き込み(作成・編集・移動・削除)は `ai_wiki/` 配下のみ**。Wiki 運用の名目で
  プロジェクト本体のコード・設定・ドキュメントを変更してはならない
- プロジェクト側の**読み取りは無制限**。Sync・Record の裏取りでは実体を積極的に読め
- 例外なし。本体側に直すべき問題を見つけた場合も、修正せず報告にとどめよ
- `WIKI_RULES.md`(この文書)と `01-Purpose.md` は人間管理であり書き込み禁止。
  Purpose の更新が必要と考えたら、Synthesize レポートで提案せよ(5.5)

## 3. ディレクトリ構造

```
ai_wiki/
├── WIKI_RULES.md        # 本規約(人間管理・LLM は読むだけ)
├── 00-Index.md          # 入口。全ページの目録+関係性マップ+タグ一覧
├── 01-Purpose.md        # 目的・主要な問い(人間管理・LLM は読むだけ)
├── 90-Log.md            # 更新ログ(時系列)
├── 99-Insights.md       # Synthesize の出力レポート(追記式)
├── concepts/            # ストック型: 検証済みの恒久知識(主題単位)
├── investigations/      # フロー型: 調査中・作業中の一時メモ(調査単位でよい)
│   └── archive/         # 退避場所(削除の代わり)
└── queries/             # Q&A の資産化(任意。必要になったら作る)
```

- どの操作でも**入口は必ず `00-Index.md`**。Wiki 全体を無差別に読み込まず、
  Index から必要なページだけを辿れ(全ページ読込が許されるのは Synthesize のみ)
- `01-Purpose.md` は「この Wiki が何を蓄積し、何を知ろうとしているか」の宣言。
  Record の取捨選択と Synthesize の分析は、これを文脈として行え
- ページの削除はしない。不要になったページは `investigations/archive/` へ
  日付プレフィックス付きで移動する(例: `2026-07-12_session_timeout.md`)
- **ページの単位**: `concepts/` は**主題型**(1概念・1対象 = 1ページ)に保つ。
  「◯月◯日の調査結果」のようなセッション型のページを concepts に置いてはならない。
  `investigations/` は調査単位(セッション型)でよく、主題型への再構成は Promote で行う(5.3)。
  知識の地図(リンク・関係性マップ)が意味を持つのは、ページが主題を表すときだけである

## 4. ページの書式

すべてのページは YAML フロントマターを持つ:

```markdown
---
type: concept | investigation | query
status: active | stale | archived
created: 2026-07-12
updated: 2026-07-12
verified_against:
  target: "src/config.py"          # 裏取りした実体のパス(複数なら配列)
  version: "abc123"                # git 管理下ならコミットハッシュ、なければ mtime
  date: "2026-07-12"               # 検証実施日
tags: [認証, API設計]              # 横断的トピック(1〜5個)
topics: [認証フロー, セッション管理]  # investigation のみ: 含まれる独立した主題
question: "..."                    # type: query のみ必須
answer: "..."                      # type: query のみ必須(回答の要旨1〜2文)
---
```

- `verified_against` は「いつ・何と突き合わせて書いたか」の痕跡。実体で裏取りできない
  思索メモの場合は `target: none` と書く
- `topics` は investigation ページに含まれる独立した主題の一覧(Record の分析ステップで
  抽出したもの)。Promote が主題単位への分割・統合を判断する材料になる(5.3)
- ページ間リンクは**相対パスの Markdown リンク**(`[題名](../concepts/xxx.md)`)を使う。
  `[[wikilink]]` は使わない(人間がこの節を書き換えて許可した場合のみ可)
- ページ末尾に `## 関連ページ` セクションを置いてよい。ただし新設・追記できるのは
  Record(自分が書くページ)と Promote(承認済み提案の反映)のときだけ(6章)
- 1ページの目安は 100〜200 行。本文中に変更履歴テーブルは置かない(履歴は 90-Log.md に任せる)

## 5. 操作セット(すべて人間の指示で発動)

指示を受けたら、どの操作に該当するかを判断して該当節の手順に従え。
複数操作の同時実行を指示された場合は Sync → Record → Promote → Lint → Synthesize の順で行え。

### 5.1 Sync — プロジェクト実体との整合(指示例: 「状態をチェックして Wiki を更新して」)

1. `00-Index.md` を読み、全ページの目録と `status` / `updated` を把握する
2. 指示でスコープが指定されていればその範囲、なければ既定スコープ
   (ディレクトリ構成・README・主要設定・直近変更ファイル)の実状態を把握する。
   git 管理下なら前回 Sync 以降の `git log` / `git diff --stat` を差分源として優先する
3. スコープに関係する各 active ページについて `verified_against` の対象と現状を突き合わせる:
   - 一致 → `updated` / `verified_against` のみ更新
   - 乖離 → まず乖離点を列挙してから、本文を実体に合わせて修正する(実体が正)
   - 対象消滅 → `status: stale` に変更し 90-Log.md に記録(archive 移動はしない。Promote で人間が判断)
4. 実体にあるが Wiki 未収載の重要要素(新モジュール・新設定)は、**ページを勝手に作らず**
   Index の「未収載の実体」節に列挙し、ページ化するかを人間に確認する
5. `00-Index.md`(ページ一覧・未収載)と `90-Log.md` を更新する

**Sync でやらないこと**: ページの新規作成、関連リンクの一括生成・書き換え、archive への移動。

### 5.2 Record — 調査結果の記録(指示例: 「ここまでの調査結果を Wiki に記録して」)

**分析→生成の二段階**で行う。いきなり本文を書き始めないこと。

1. **分析**: `00-Index.md` と `01-Purpose.md` を読み、次を列挙する —
   ①記録内容の要点、②含まれる独立した主題(→ `topics` に記録する)、
   ③主題ごとの既存ページの有無(あれば新規作成せず追記を提案する)、
   ④既存ページとの接続と矛盾。矛盾を見つけたら、隠さずページ本文に明記する
2. **生成**: `investigations/` に新規ページを作成する。**既定は1調査1ページ**だが、
   手順1で互いに独立した主題が複数見つかった場合は主題ごとにページを分けてよい
   (その場合も各主題について既存ページとの重複を確認すること。細かい断片の
   大量生成はしない)。書く前に、可能な限り実体を読んで裏取りし、
   `verified_against` を正しく埋める。`tags` を1〜5個、`topics` を付ける
3. 手順1で列挙した接続を `## 関連ページ` に書く(網羅は不要)
4. `00-Index.md` のページ一覧と `90-Log.md` を更新する

**Record でやらないこと**: `concepts/` への直接作成(昇格は Promote 経由)、既存ページの書き換え。

### 5.3 Promote / Organize — 整理と昇格(指示例: 「一区切りついたので Wiki を整理して」)

1. `investigations/` の各ページを評価し、人間に提案する:
   - 恒久価値がある(実体で検証済み・今後も参照される)→ `concepts/` へ昇格。
     **昇格はファイル移動ではなく主題単位への再構成である。** ページの `topics` と
     本文から次のいずれかを提案する:
     (a) 複数の主題を含む → 主題ごとに新しい concept ページへ**分割**
     (b) 既存 concept と同じ主題 → その concept ページへ**統合**(追記・更新)
     (c) 単一の新しい主題 → 主題名のページとしてそのまま移動
     いずれの場合も concepts は主題型(1概念 = 1ページ)を保ち、
     セッション型のページ名(「◯月◯日の調査」等)を concepts に持ち込まない
   - 用済み・古い → `investigations/archive/` へ日付プレフィックス付きで移動、`status: archived`
   - 継続中 → そのまま
2. 人間の承認を得てから再構成を実行する。分割・統合の元になった investigation ページは
   archive へ移動し(日付プレフィックス)、末尾に行き先 concept ページへのリンクを残す。
   移動・再構成に伴うリンク切れを修正する
3. `99-Insights.md` に未反映の「関連性の提案」があり人間が承認した場合、
   該当ページの `## 関連ページ` と Index の「関係性マップ」に反映する
4. Index のタグ一覧を全 active ページの `tags` から再集計する
5. `00-Index.md` と `90-Log.md` を更新する

### 5.4 Lint — Wiki 内部の健全性チェック(指示例: 「Wiki の健全性をチェックして」)

以下を検出し、**報告と修正提案のみ**を行う。人間の承認なしに修正しない:

- リンク切れ(存在しないページへの相対リンク)
- 孤立ページ(どこからもリンクされず、関連ページも持たない active ページ)
- 重複・類似の概念ページ(統合候補)
- ページ間の矛盾する記述
- 長期 stale(`status: stale` のまま放置、または `updated` が古い active ページ)
- タグの乱れ(表記ゆれ・一度しか使われないタグの乱立)
- 肥大化(200行超のページ、200行または50ページ超の Index → 分割を提案)

archive 配下は Lint の対象外。

### 5.5 Synthesize — 総合洞察の抽出(指示例: 「知見を総合して新たに見えるものを抽出して」)

**既存ページは一切書き換えない。出力先は `99-Insights.md` への追記のみ。**

1. `01-Purpose.md`(目的・主要な問い)を読んで分析の文脈とし、`00-Index.md` から
   全 active ページ(concepts / investigations / queries)を把握する。
   指示でスコープ指定があればその範囲に絞る
2. 各ページの本文を読み、次の観点で横断分析する:
   - **共通パターン**: 複数ページに繰り返し現れるテーマ・課題
   - **矛盾・対立**: ページ間で食い違う主張や前提
   - **トレンド**: 時間とともに変化している関心・方針(90-Log.md も参照)
   - **未検証領域(ギャップ)**: 言及されるが深掘りされていないトピック、および
     01-Purpose.md の主要な問いにまだ答えられていない領域
   - **新たな仮説**: 複数の知見の組み合わせから初めて導かれる示唆(要検証と明記)
   - **関連性の提案**: 張るべきなのに張られていないページ間リンク
   - **Purpose 更新の提案**(あれば): 蓄積された知見と 01-Purpose.md の記述に
     ズレが生じていたら、更新案を書く(反映するかは人間が判断)
3. `99-Insights.md` に日付セクションとして**追記**する(雛形は同ファイル冒頭に記載。上書き禁止)
4. レポート末尾に「Record(新規調査)や Promote(リンク反映・整理)を実行するかは
   人間が判断する」と付記し、90-Log.md に実行記録を1行追記する

## 6. 関連リンクと関係性マップの更新規則(腐敗対策)

ページ間の関連情報を書き換えられる契機を次の3つに限定する:

1. **Record**: 自分が新規作成するページに、把握している関連リンクを付ける
2. **Synthesize**: 関連性を「提案」として 99-Insights.md に出す(ページには書かない)
3. **Promote**: 人間が承認した提案だけを、各ページの `## 関連ページ` と
   Index の「関係性マップ」に反映する

Sync や Lint のついでに関連リンクを生成・書き換えることを禁止する。

## 7. ログの書式

`90-Log.md` には操作のたびに1行追記する:

```
- 2026-07-12 | Sync | concepts/arch.md 更新(src/ 構成変更を反映), investigations/xxx.md を stale 化
```

日付・操作名・変更対象・理由(簡潔に)を含めること。

## 8. その他の規則

- **機密情報の禁止**: API キー・シークレット・トークン・個人情報を Wiki に書かない。
  実体側で見つけても Wiki に転記しない
- **git コンフリクト**: Wiki が git 管理されている場合も、コンフリクト解決は人間が行う。
  LLM はマージ作業に関与しない
- **queries/ の運用**(任意): 人間の質問に調査して答えたとき「これを資産化して」と
  指示されたら、`queries/` に `type: query` のページを作る(`question` / `answer` 必須)。
  ディレクトリが無ければそのとき作る
- **迷ったら聞く**: 規約にない判断(大量のページ新設、構造変更など)は実行前に人間に確認する
